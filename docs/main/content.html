Creating & Formatting Responses

Botpress includes its own Content Management System (or CMS) to manage all the content your chatbot will display to the user during a conversation. Everything your chatbot says comes from the CMS. Before discussing how you can create and edit content, we should understand the CMS's different concepts in Botpress.

Content Type

The content type defines the structure of what your chatbot sends. It also establishes how Botpress should render the content. For instance, a Content-Type could be text or an image. It can also be complex, for instance, a carousel of images or a password authentication form. They can describe anything depending on what you use your chatbot for.

üåü Tip: As a general rule, the more domain-specific the Content Types are, the easier it is to manage the bot for non-technical people.

Botpress comes with the following default content types:

  • Card A card message with a title with optional subtitle, image, and action buttons.
  • Action Button A button that triggers an action, often used in cards
  • Carousel An array of cards
  • Image A message showing an image with an optional title
  • Single Choice A suggestion of choices to the user who then has to pick only one (with an optional message)
  • Text A regular text message with optional typing indicators and alternates
  • Dropdown Displays a list of options to the user

Botpress provides an interface to add and manage content. The developer may also optionally add content via the visual editor interface. Below is an overview of both methods of creating content.

The first method is to use the Content interface. In practice, this method is mostly used to edit the content that has already been added to the chatbot. When building a chatbot, developers normally add content while constructing conversation workflows. They usually ignore the finer details like spelling, grammar, tone, picture quality, and links.

A content manager then comes along and reviews the developer's content, making it more product/use-case agnostic and aligning it to the character of the chatbot.

As mentioned earlier, another way is to add content using the Flow Editor interface. This way, a chatbot developer can add content as they build conversation workflows, making it easier to test out the chatbot as they go.

Content Types are particular to the bots they are associated with. Here are some typical examples:

  • The built-in "Text" content-type defines how text is rendered (using markdown).
  • A restaurant "Menu" and "MenuPage" types may include descriptions of ingredients, health information, and links to payment gateways.
  • A "QuestionWithChoices" type which may include further drill-down after choosing certain choices.
  • An "ImportantBroadcast" type may also have capabilities to add video and iframes to pertinent websites.

As you can see, Content Types on Botpress are much more specific than generalized "message types" on traditional bot building platforms.

Developers define content Types in JavaScript. Each Content Type has its own .js file, and Botpress automatically finds and registers new Content Types based on the directory and naming convention of the file.

Botpress also supports the REACT user interface building framework. This means that you can create a REACT component (e.g., a date-picker content element with a calendar.) and render it in the webchat as a content type.

It is important to keep in mind the channel where your chatbot will live because it may not always be possible to render some content on all channels.

Content Element

A Content Element contains the data of a Content-Type. Multiple Elements can belong to a single Content-Type. For instance, the "text" Content-Type will contain an Element for every text response from your chatbot, e.g., "Hello!", "What is your name?" etc.

Here's a Content Element example:

JSON

  {
    "id": "builtin_text-m137O6",
    "formData": {
      "markdown$en": true,
      "typing$en": true,
      "text$en": "Hie!"
    },
    "createdBy": "admin",
    "createdOn": "2020-11-24T06:39:16.479Z",
    "modifiedOn": "2020-11-24T06:39:16.479Z"
  },

All Content Elements of the same Content Type are stored within a single .json file under the data/bots/{your-bot}/content-elements/ directory.

Tip: Remember that a Content Type tells how content gets rendered and a Content Element tells what to render.

Channel Specific Rendering

All Content Types define a renderElement function that tells how a Content Element gets rendered on different channels.

Note: This is critical because every channel is different and has a different set of functionalities. You want to customize and leverage the different platforms' features to offer the best user experience possible.

Rendering Function

Here's the web rendering function of the Text Content Type:

function renderForWeb(data) {
  const events = []

  if (data.typing) {
    events.push({
      type: 'typing',
      value: data.typing
    })
  }

  return [
    ...events,
    {
      type: 'text',
      markdown: true,
      text: data.text
    }
  ]
}

Rendering to Multiple Channels

The default Content Types are already pre-configured to work on all channels which are supported out-of-the-box.

Now, if we'd like to render for the messenger channel, we would add a specific rendering function for messenger and call it when channel === 'messenger'

function renderForMessenger(data) {
  const events = []

  return [
    ...events,
    {
      type: 'message',
      user: data.profile,
      text: data.text,
      raw: data
    }
  ]
}

function renderElement(data, channel) {
  if (channel === 'web') {
    return renderForWeb(data)
  } else if (channel === 'messenger') {
    return renderForMessenger(data) // We add our rendering function
  }

  return []
}

Translation

Pro-licence holders can create Multi-lingual bots on licensed installations. For more details on licensing, see the Pro Features section.

Your bots can support multiple languages. On the first interaction, the bot extracts language from the browser and saves it as a user attribute (available on the event as user.language). If the language extracted from the browser is not configured on the bot, it will use the default language.

Please note that once the language property is set, it will never be overwritten. Therefore, it is advisable to ask the user their preferred language or use the NLU engine to detect it.

When rendering content elements, we will try to render the user's configured language. If this fails, conversations will be in the bot's default language.

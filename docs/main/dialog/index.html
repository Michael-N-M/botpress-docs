Creating Conversations

Botpress uses a Dialog Engine to handle conversations. This engine is responsible for user interaction with a bot from the user input to the bot response. Between the two, a whole lot is happening, which we will delve into here.

Overview

The Dialog Engine uses Flows, which represent the overall conversation workflow. For instance, giving feedback on a product can be contained within a single flow.

A flow is composed of Nodes which execute a series of Instructions. These instructions can be content (When a bot says something to the user like "Hie" or displays a Content Element) or an Action (code with a specific purpose, e.g., saving what the user said in the database).

There are two sources of actions, namely:

-Bespoke Actions These are actions that execute Javascript code written by the bot developer. -Built-In Actions Botpress comes pre-configured with these actions. They are automatically installed you run Botpress.

Nodes

Nodes are the primary units of the conversation logic of your bot. An active conversation (which we call a "session") always has one and only one active node at any given time. A node generally transitions to another node or flow. So, in essence, conversations are a series of nodes and flows strung together.

The conversation ends when there is no flow or node to connect to next or where the user doesn't interact with the bot for an extended period. In both instances, a message from the user will start an entirely new session.

It's important to note that every bot comes with a flow named Main by default. The main flow is always the first to execute.

A node is separated into three different stages: On Enter (A), On Receive (B) and Transition (C).

The chatbot also executes flow-wide actions, meaning they are checked every time the bot moves to the next node. Flow-wide On receives (D) Flow-wide Transitions (E)

Node Lifecycle

On Enter

onEnter is a list of instructions executed when the node is begun. If multiple instructions are defined, all of them will be executed sequentially. Instructions can be in the form of content (like text) or actions (Javascript coded instructions).

On Receive

onReceive is a list of instructions executed when an active node receives a message. If an action is defined at this stage, the node will automatically wait for user input (orange node).

When this property is left unused, the node is non-blocking, which means it will flow straight from onEnter to the Transitions.

Transitions

Transitions are conditions that are evaluated only after onReceive or onEnter have been executed. They define the next node or flow to be executed.

A Transition always has a target that we call a Destination. The target can be:

  • A different Node
  • A different Flow
  • The previous Flow (see Flow Stacks)
  • Itself (Loopback on itself)
  • The end of the conversation

Special cases: If no condition is defined, the default behavior is that the conversation ends. If there are conditions defined but none match, nothing happens, i.e., the current node stays active, and it will flow when a condition is matched. By default, the onNext will only be retried after onReceive is re-invoked.

Flow-wide On Receives

You can define an onReceive instruction that will always be executed before every node's onRecieve.

üëì Examples: Flow-wide On Receive

  • Audit Trail: Record specific logs of messages received in the scope of this particular flow
  • Authentication Gate: Run some authentication
  • Sentiment Analysis: Making sure the sentiment of the conversation is staying healthy

To define new _Flow-wide On Receive Actions _, navigate to the relevant flow, then double click anywhere on the checkered background to show the Flow Properties Pop up. You can also click on the links in the top left corner of the flow editor. Under the On Receive section, click the Add Action button to add a new action.

Flow-wide Transitions

A Flow-wide Transition instruction allows you to override node transitions when the condition is successful.

üëì Examples: Flow-wide Transitions

  • Authentication Gate: Re-route the user to the login flow if they are not authenticated
  • Sentiment Analysis: Re-route the user to the human fallback node if the conversation is degrading
  • Matching flow-wide intents such as "cancel" etc.

Flows

More complex bots are broken down into multiple smaller flows instead of creating one big flow. Bot developers use multiple flows to ease maintainability and reusability.

Flow Lifecycle

A flow always starts at the startNode of its *.flow.json file. The start node indicates the name of the node on which to start. Once the node is selected, the Dialog Engine will enqueue the active node's instructions (more on instructions later). Then it will process the instructions sequentially. The Dialog Engine is event-based and is non-blocking by default, which means that a flow will execute all it can execute until it is instructed to wait.

Note: There are currently two reasons for a flow to "wait":

  • A node is marked as waiting for user input
  • A node couldn't match a condition to transition to another node

Once the first node is processed, the Dialog Engine will proceed to the next node in the flow until it reaches the very end. Flows are pretty straight forward. Nodes also have a lifecycle of their own. It is the nodes that do the heavy lifting in a flow. The flow only orchestrates them.

Flow Storage

Flows are stored as JSON files in the bot's source files. The directory route for flows is data/bots/<name of bot>/flows/ folder. Each flow is split into two files: the logic (*.flow.json) and the visual-specific properties (*.ui.json). The reason to split these is to make it easier to maintain and review changes.

  • *.ui.json files can almost always be ignored from code reviews as they don't affect the bot's functionality.
  • *.flow.json files could also, in theory, be created manually by developers instead of using the GUI.

State

Each conversation has a State associated with it. The state is created when the conversation session starts and is destroyed when the session ends. See Flow Lifecycle.

Note: The state is global to the conversation, meaning it's not restricted to a single flow. In other words, if the conversation goes through more than one flow, the state is shared for all the flows.

Session Timeout

The Dialog Engine waits for an inactive user's input, but the session will Timeout after a while.

Timeout Uses

Timeout allows you to end the conversation gracefully if need be. It can also be useful to do some processing before deleting the session. Some common actions performed at session timeout are:

-Saving the user's contact information to an external database. -Telling the user how to contact you. -Informing the user his session has timed out. -Checking out a shopping cart on e-commerce bots. -Showing a product promotion content element. -Recap of a lesson on Learning Management System integrated bots

Timeout Flow

You can use a Timeout Flow to handle your timeout logic. It works just like a regular flow. All you have to do is add a Flow called timeout.flow.json to your bot and specify a start node. The Dialog Engine will detect your Timeout Flow next time a user times out.

Timeout Node

Like the Timeout Flow, the Timeout Node should be called timeout and should belong to the current flow. You don't have to link it to any other node. The Dialog Engine will detect the node based on its name.

Timeout Transition

Another option that requires some coding is to add the property timeoutNode to your *.flow.json file. Thereafter, assign it to the node that should handle the timeout. It can be any node, and it doesn't need to be called a particular way.

‚ö† Important: Once the Dialog Engine has processed the timeout, it will delete the session.

Actions

An Action is JavaScript code that is executed in a Node.js Virtual Machine. Actions can do anything you want, including the following common uses: -Call an API -Store to or retrieve data from a database. -Link to a payment gateway. -Carry out a mathematical calculation.

Actions are called in the onEnter or onReceive stages of the flow lifecycle. There are two types of Actions:

  • Script: A user-defined Action that is used to run custom code.
  • Output: An output Action that is used to make a bot output something.

To learn more on Actions, please refer to the Custom Code section.

Botpress comes with a built-in set of actions. For learning purposes, we will examine the most basic action called Wait. It allows the developer to pause the bot before running the next instruction. For example, when the bot is about to present a long text message, the developer may use the wait action to simulate real typing. 

  * Wait for a bit.
 *
 * @title Wait/Delay
 * @category Utility
 * @author Botpress, Inc.
 * @param {number} delay=1000 - The number of milliseconds to wait
 */
const wait = async delay => {
  return new Promise(resolve => setTimeout(() => resolve(), delay))
}

return wait(args.delay || 1000)

As you can see, the action is just a simple asynchronous arrow function that takes the number of milliseconds to delay as a parameter. When building an action, you can (and indeed should) specify the action type, its category, the author, and parameters.

Skills

After building a couple of flows/bots, you'll quickly notice that there are some common patterns that you find yourself implementing over and over. Skills come to the rescue.

Skills are dynamic flow generators. They can be seen as reusable components between multiple flows or even multiple bots.

Installing Your Own Skills

A module must expose a skill. Modules can host any number of skills. All you need to do is install the required module, and you will have access to its skills.

Using Built-In Skills

Skills can be accessed from the Botpress Flows GUI. After installing a skill module, navigate to a flow in the Graphical Flows Editor, then locate the Skill in a middle right tab (see A below) or by right-clicking anywhere in the Botpress Flows GUI: After filling in the form, you'll be able to click anywhere in the flow to insert the skill to be consumed by the other nodes.

Persistence

Skills are stored as flows under the data/bots/your-bot/flows/skills folder. You can also visualize the generated skills from the GUI:

Editing skills

Once a skill node has been generated, you may click on that node and click "Edit" on the left panel to edit that node, which will update the generated flow automatically behind the scenes.

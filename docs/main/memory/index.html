Storing Information

Accessing System Variables

To remember information supplied by the user, such as names, dates, answers to questions, and extracted slots, Botpress has a system for storing variables during the course of a conversation.

When you engage a Bot, Botpress tracks all the associated variables and parameters as the conversation transitions from one state to another. A tree of all the system parameters being followed is shown in the conversation emulator (see A below). You can access these system parameters from the flow builder and within your code (including in Actions). To do so, all you need to do to reference a parameter by prefixing the path shown in the emulator with "event.".

For example, the emulator's path to the language parameter is nlu.language. You can reference that parameter by adding "event" to the emulator's path, i.e., event.nlu.language (see B above).

You can access system parameters by enclosing them within two sets of curly brackets in the Flow Editor. In a message composed as a text content element, you could say: The bot's language is {{event.nlu.language}}.

You can also set variables to be the value of a system parameter, as illustrated in this article's Built-In Memory section.

For raw expressions or code (such as in Actions), you don't need to include curly brackets.

In the same way, as described above, it would be possible to access the values of extracted slots by copying the path from the emulator and prefixing it with "event." i.e. {{state.session.slots.food.value}} in the flow builder and state.session.slots.food.value in code. "food" is a slot that was set in an intent by the bot builder.

It is also possible to access the parameters using the syntax {{state.session.slots["food"].value}}

The most widely used system parameter is event.payload.text. It returns the latest text response from the user.

Dialog Memory

Botpress lets you store data in many ways thanks to in-built Actions, which allow you to set and retrieve variables.

Also, Botpress supports the Knex query builder, which is handy for further data analysis and connection to external data sources.

The Set Variable Action

The Dialog Memory is how your Bot will remember things in the context of a conversation. To set a variable using the Set Variable action, you will need to supply the type, name, and value.

Type

There are four different kinds of memory in Botpress: user, session, temporary, and bot memory. These memory types differ only in duration and scope. The value of type in the Set Variable user interface must be set to one of these four types.

Name

Botpress allows you to give the variable a name, which you will use every time you call, change, or reference it. It is recommended to use camelCase or pascal_case when naming variables.

Value

This is the data stored in the variable. You can set a system parameter as the value by enclosing it within two sets of curly brackets, for instance, {{event.user.text}}

User Memory

Variables set using the user namespace will be saved as attributes for the user. This means that those attributes will always follow the user, notwithstanding conversations or time period.

When a user sends a message to the Bot, the first middleware is tasked with loading that user's information. After everything is processed, any changes to the user object will be persisted to the database. This means that you can alter the user object by utilizing middlewares and actions.

User Memory - Data Retention

Since privacy is an important matter, a built-in system makes it dead-easy to set retention periods for different information types. For example, you could have a policy that says email expires after 2 months or remember user's mood for 1 day. Whenever the user's attribute is changed, the expiration policy is updated.

Here's how you could configure such a policy:

//data/global/botpress.config.json

dataRetention: {
  janitorInterval: '2m', // Check each 2 mins for expired data
  policies: {
    email: '60d', // Keep email for 30 days, reset if it is changed
    mood: '1d' // Forget user's mood after 1 day
    someChoice: '5m' // Keeps the variable alive for 5 minutes
  }
}

Session Memory

The session store is alive for the duration of the user's session. How long is that? Well, it depends on the value of sessionIntervalTimeout in botpress.config.json.

This is also where we keep the last messages sent by the user. This information is used by the Decision Engine to understand the user's will better and to avoid repeating meaningless stuff.

Temporary Memory

The temp memory is the most heavily used type of memory in Botpress. To keep it simple, this memory is kept from the beginning of flow until the end. As long as nodes are linked together, it will be available. If you were a Botpress 10.x user, this was better known as the state of the dialog.

Common use cases include calling an action, saving the result in the temp memory, then send a content element including the answer to the user.

For example, you want to list your servers' name, which should be fetched from an API.

This would be your action, fetch_servers.js:

const listServers = async () => {
  try {
    const { data } = await axios.get(`https://mysite.com/servers`)
    temp.servers = data.servers.join(', ')
  } catch (error) {}
}

return listServers()

That action would fetch the name of your servers. Then you could send a content element similar to this:

Here's the list of our servers: {{temp.servers}}

As you can see, it's effortless to use!

Bot Memory

The bot memory is the same value for all users of the Bot. Think of it as a global variable, but scoped to this Bot only.

Variables In Your Code

You can create variables within Actions and Hooks. You can also set new values to variables within your code.

For example, if you set temp.user_name = "John", the variable temp.user_name will be created and set to the value "John."

Below is a code example where a system variable is used. On line 12, a check is carried out to test whether English is the language in use.

/**
 * Sends feedback (1 for positive or -1 for negative) at the end of a goal
 *
 * @title Send Feedback
 * @category NDU
 * @author Botpress, Inc.
 * @param {number} [value] - The feedback value. Use 1 for positive feedback, -1 for negative feedback
 */

const sendFeedback = async value => {
  const feedback = parseInt(value.trim())
  if (feedback !== 1 && feedback !== -1 && event.nlu.language == 'en') {
    throw `Unexpected value: ${value}`
  }

  await bp.events.saveUserFeedback(temp.eventId, event.target, feedback, 'workflow')
}

return sendFeedback(args.value)

As you can see, in code or raw expressions, the reference to the variable does not need the double curly brackets.

Changing Memory Contents

There are two different ways to edit these 4 different types of data. The simplest way is to use the action setVariable. You only have to specify the type of memory, the variable's name, and its value.

Another common way is using actions. Actions allow you to edit these variables directly. For example, you could write `user.firstname = 'potato" in your code file to update the user's name.

Please check out the Custom Code section for more details about this.

General Storage

What if the four previous types of storage don't fulfill my requirements? Don't worry, there are other options!

The Key-Value Store is a general-purpose store that allows you to store any data. You will need to manage expirations and data update yourself since Botpress will not update these values independently.

The KVS is available from the Botpress SDK
